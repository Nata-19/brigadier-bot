# SPEC.md — brigadier-bot («Помощник бригадира»)

> Спецификация Telegram-бота. Это техническое задание на код, не промпт для LLM.
> База — инструкция, написанная под Эврика, очищенная от всего, что относится к LLM-поведению.

---

## 1. Назначение

Бот учёта дневных работ бригадиров на виноградниках. Бригадир вводит сообщение фиксированного формата → бот парсит, валидирует, считает по инвентаризации, сохраняет в SQLite, выдаёт отчёт.

Канал: Telegram. Стек: Node.js + Telegraf + SQLite + xlsx. Деплой: VPS, systemd-сервис.

---

## 2. Роли и доступ

Три роли:

- **Админ** — управляет всеми пользователями, видит данные любого главного и любой бригады.
- **Главный бригадир** — имеет **свой набор кварталов** в инвентаризации (см. §10). Под ним работает одна или несколько бригад со своими бригадирами. Видит сводные отчёты по всем своим бригадам.
- **Бригадир** — рядовой пользователь, на земле руководит конкретной бригадой. Привязан к главному (поле `head_id` в `users`). Ведёт учёт своей бригады, видит только свои записи.

Жизненный цикл (для главных и обычных бригадиров одинаков): `pending` → `active` → `blocked`. Активацию делает админ.

## 3. Команды

**Бригадир (рядовой, под главным; только в статусе `active`):**
- `/start` — регистрация (попадает в `pending`, главному + админу уведомление)
- `/work <вид> [main]` — открыть вид работ. С флагом `main` — назначить основным (см. §4.1, §4.5)
- `/works` — показать список открытых видов
- `/end <вид>` — закрыть конкретный вид (закрытие только бригадиром, не автомат)
- `/end` — без параметра: если открыт один — закрыть; если несколько — спросить какой
- `/send <кто> → <куда/на что>` — запись отправки без учёта (см. §4.3, §5.4)
- `/today` — дневной отчёт за текущую дату
- `/period DD.MM-DD.MM` — отчёт за период (с разбивкой по видам работ)
- любое сообщение в формате ввода (см. §5) → принимается как запись

**Главный бригадир:**
- Все команды рядового бригадира (если §2.1 решит, что главный тоже пишет записи)
- `/team` — список своих бригадиров со статусами
- `/team-today` — сводный дневной отчёт по всем своим бригадам
- `/team-period DD.MM-DD.MM` — сводный отчёт за период по всем своим бригадам
- `/brigadier-data <user_id> [период]` — данные конкретной своей бригады

**Админ:**
- `/approve <user_id> [head_id]` — активировать пользователя. Без `head_id` — это сам главный. С `head_id` — это бригадир под указанным главным.
- `/block <user_id>` — заблокировать
- `/list` — все пользователи со статусами и ролями
- `/alldata <user_id> [период]` — данные конкретной бригады
- `/reload-inventory <user_id>` — перезагрузить инвентаризацию для главного из его Excel

## 4. Виды работ

Закрытый список валидных видов работ:

1. обрезка
2. подвязка сухая
3. обломка
4. заводка
5. цаповка
6. опрыскивание
7. посадка
8. полив
9. установка приштамбового колышка
10. установка фиксаторов
11. уборка винограда

### 4.1. Модель «список открытых видов работ»

У каждого бригадира — **список открытых видов работ** (от 0 до N). Открытые виды держатся днями, бригадир добавляет и убирает их вручную:

- `/work <вид>` — добавить вид в список открытых. Можно открыть несколько видов одновременно: один основной + дополнительные.
- `/end <вид>` — закрыть конкретный вид. **Закрытие — только бригадиром, никогда не автоматическое.**
- `/works` — показать список открытых видов.

**Понятие «основной» vs «дополнительный»:** «основной» вид назначается **явно** через флаг `main` при открытии: `/work обрезка main`. В каждый момент основной может быть только один. Если бригадир открывает новый вид с флагом `main`, прошлый основной становится дополнительным. Если без флага — вид добавляется как дополнительный.

Если бригадир пишет запись учёта без явного указания вида (`[вид]`) — берётся основной. Если хочет указать другой — ставит `[вид]` в записи (он должен быть в списке открытых).

**Если основной не назначен** (например, открыт только один вид без `main`) — он считается основным по умолчанию. Если открыто несколько без `main` — ошибка при записи без `[вид]`: «Назначь основной вид: /work <вид> main».

**Ввод без открытых видов:** бот отвечает предложением открыть (`Сначала открой вид работ: /work <вид>`).

**Открытие вида, который уже открыт** — игнорируется или сообщение «Вид уже открыт».

### 4.2. Указание вида в записи учёта

Если у бригадира открыто несколько видов — в записи можно явно указать, какой вид у конкретной клетки или сотрудника. Метка `[вид]` ставится:
- **На уровне клетки:** `Квартал N, Клетка N [подвязка сухая]: ...` — вся клетка на этом виде
- **На уровне сотрудника:** `Сидоров [подвязка сухая] - с 1 по 2` — отдельный сотрудник на другом виде, остальные в клетке — на основном или указанном на клетке

Если `[вид]` не указан — берётся **основной** (первый открытый).

**Указанный вид должен быть в списке открытых** — если нет, бот возвращает ошибку с подсказкой открыть его.

Пример (открыты: обрезка, подвязка сухая; основной = обрезка):
```
Дата 2026-04-23:
Квартал 1, Клетка 3: Сидоров - с 2 по 4; Иванов [подвязка сухая] - с 5 по 7
Квартал 1, Клетка 4 [подвязка сухая]: Петров - с 1 по 3; Маметов - с 4 по 5
```
- Сидоров → `обрезка` (основной)
- Иванов → `подвязка сухая` (явно)
- Петров и Маметов → `подвязка сухая` (на клетке)

### 4.3. Отправка (без учёта рядов/кустов)

Отдельный тип записи: бригадир фиксирует факт, что часть людей **отправлена на работу, которую нельзя посчитать обычным способом**:

- Вид работы **вне закрытого списка** §4 (например, мульчирование, ремонт оборудования, помощь соседней бригаде)
- Или другой квартал/клетка **без точного указания рядов**
- Или просто «N человек куда-то отправлены»

**Свойства записи отправки:**
- Указывается: количество людей или фамилии, и куда/на что (свободный текст)
- **НЕ** идёт расчёт по инвентаризации (нет рядов → нет кустов)
- Сохраняется отдельно (поле `entry_type = "dispatch"`), но привязана к дате и бригадиру
- В дневном отчёте — выводится отдельным блоком (см. §7.4)

Формат ввода — см. §5.4.

### 4.4. Решения по открытым вопросам (закрыто 2026-05-08)

- **Несколько видов одновременно** — да, может быть, бригадир сам управляет списком. Один основной (первый открытый) + дополнительные.
- **Открытие нового вида** — **не закрывает** старый автоматически. Закрытие только командой `/end <вид>`.
- **Ввод без открытого вида** — ошибка с предложением открыть.
- **Режим «только ряды»** — нет автоматических привязок к виду. Бригадир сам говорит фразой `только ряды` или `без кустов` в сообщении.
- **Отправка** — может содержать число людей или фамилии (любое из двух).
- **Отправки в `/period`** — да, входят отдельным блоком (см. §7.4). Без них отчёт за период будет неполным — это была одна из причин неудачи прошлой попытки бота.

### 4.5. «Основной» вид назначается явно (вариант B)

- `/work обрезка main` — открыть «обрезку» как основной
- `/work подвязка сухая` — открыть «подвязку» как дополнительный
- `/work опрыскивание main` — теперь основной = «опрыскивание», «обрезка» автоматически становится дополнительным
- В записи учёта без `[вид]` — берётся основной

## 5. Формат ввода

### 5.1. Управление видами работ

```
/work обрезка main          # открыть «обрезку» как основной
/work подвязка сухая        # добавить дополнительный вид (без флага main)
/work опрыскивание main     # переключить основной на «опрыскивание»
                            # «обрезка» становится дополнительным

Начинаем: обрезка           # текстовый эквивалент `/work обрезка main`
                            # (если ничего не открыто — назначается основным)

/works                      # список открытых: основной отмечен ⭐

/end подвязка сухая         # закрыть конкретный вид
/end                        # закрыть один открытый (если он один); иначе спросить
```

Закрытие — **только бригадиром**. Открытие нового вида не закрывает старые. Если закрыт основной и остались дополнительные — бот спрашивает кого сделать основным.

### 5.2. Стандартный ввод учёта

```
Дата YYYY-MM-DD:
Квартал N, Клетка N: Фамилия - с X по Y, A; Фамилия - с M по N
Квартал N, Клетка N: ...
```

**Правила:**
- Дата обязательна, в начале сообщения
- Несколько клеток допускаются (каждая с новой строки)
- Разделитель сотрудников внутри клетки: `;`
- Диапазон рядов: `с X по Y` (включительно)
- Перечисление рядов: через запятую
- Все значения — целые числа
- Поле `work_type` в записях заполняется так:
  - Если в блоке клетки или у сотрудника есть `[вид]` — этот вид (должен быть в списке открытых)
  - Иначе — **основной открытый вид** (первый по времени открытия)
- Если открытых видов нет — ошибка `Сначала открой вид работ: /work <вид>`

**Пример (открыта обрезка, основной):**
```
Дата 2026-04-23:
Квартал 1, Клетка 3: Сидоров - с 2 по 4; Иванов - с 5 по 7
Квартал 1, Клетка 4: Сидоров - с 2 по 4; Иванов - с 5 по 7; Петров - с 1 по 2
```
→ все записи получают `work_type = "обрезка"`.

### 5.3. Совмещение нескольких открытых видов в одном вводе

Когда открыто >1 вида, метка `[вид]` указывает, какой именно вид у клетки/сотрудника:

**На уровне клетки:**
```
Квартал N, Клетка N [подвязка сухая]: Фамилия - с X по Y; Фамилия - ...
```

**На уровне сотрудника** (внутри одной клетки часть бригады на одном виде, часть на другом):
```
Квартал N, Клетка N: Фамилия - с X по Y; Фамилия [подвязка сухая] - с M по N
```

Указанный `[вид]` **должен быть в списке открытых** (см. `/works`). Иначе — ошибка с подсказкой.

**Пример совмещения (открыты: обрезка [основной], подвязка сухая):**
```
Дата 2026-04-23:
Квартал 1, Клетка 3: Сидоров - с 2 по 4; Иванов [подвязка сухая] - с 5 по 7
Квартал 1, Клетка 4 [подвязка сухая]: Петров - с 1 по 3; Маметов - с 4 по 5
```
- Сидоров (Клетка 3) → `обрезка` (основной)
- Иванов (Клетка 3) → `подвязка сухая` (явно)
- Петров и Маметов (Клетка 4) → `подвязка сухая` (на клетке)
- Список открытых видов **не меняется**.

### 5.4. Отправка (запись без учёта рядов/кустов)

Когда часть людей отправлена на работу, которую невозможно учесть обычным форматом (см. §4.3) — отдельный тип записи.

**Команда:**
```
/send <количество или фамилии> → <куда/на что> [— <дата>]
```

**Текстовый формат:**
```
Отправил: <количество или фамилии> → <куда/на что>
```

**Примеры:**
```
/send 2 человека → Квартал 5, Клетка 2 — мульчирование
/send Сидоров, Петров → помощь соседней бригаде
Отправил: 3 человека → ремонт инвентаря
Отправил: Иванов → Квартал 7, Клетка 1 (ряды не считаем)
```

**Что попадает в БД:**
- `entry_type = "dispatch"`
- `worker_count` (число) или `worker_names` (список)
- `note` — текст после `→`
- `work_type = NULL` (вне списка)
- `rows = []`, `bushes = 0`
- `date` = текущая дата (или указанная в команде/тексте)

Отправка **не участвует** в расчётах рядов/кустов, но выводится в дневном отчёте отдельным блоком (см. §7.4).

## 6. Алгоритм обработки

1. Принять ввод
2. Распарсить дату
3. Разделить на блоки по клеткам
4. Для каждой клетки — определить квартал и клетку, выделить сотрудников
5. Для каждого сотрудника — распарсить диапазоны и перечисления, сформировать список рядов
6. Проверить дубли и пересечения **внутри ввода** (один ряд не может быть у двух сотрудников одной клетки)
7. Проверить пересечения **с историей** (тот же ряд в той же клетке за ту же дату — недопустимо для других бригадиров; для своих — также недопустимо в текущем дне)
8. Сверить рядa с инвентаризацией (квартал/клетка/ряд должен существовать)
9. По инвентаризации получить количество кустов в каждом ряду
10. Рассчитать: количество рядов и сумму кустов на сотрудника, на клетку, на день
11. Сохранить запись в БД (с привязкой `user_id`, `дата`, `квартал`, `клетка`, `сотрудник`, `ряды[]`, `кусты`, `вид_работы`)
12. Сформировать ответ (см. §7)

## 7. Формат ответа

### 7.1. Стандартный (после успешного ввода)

```
Дата: YYYY-MM-DD

Квартал - N, Клетка - N
Фамилия - X рядов - Y кустов
Фамилия - X рядов - Y кустов
Всего: X рядов - Y кустов

Квартал - N, Клетка - N
...
Всего: X рядов - Y кустов

ВСЕГО ЗА ДЕНЬ:
Ряды: X
Кусты: Y
```

### 7.2. Режим «только ряды»

Активация: ввод содержит фразу `только ряды` или `без кустов`.
Логика та же, но в ответе кусты не выводятся.

```
Дата: YYYY-MM-DD

Квартал - N, Клетка - N
Фамилия - X рядов
...
Всего: X рядов

ВСЕГО ЗА ДЕНЬ:
Ряды: X
```

### 7.3. Блок «Отправки» в дневном отчёте

Если за день были записи отправки (см. §4.3, §5.4) — добавляется отдельный блок **после** учёта по клеткам и **перед** строкой `ВСЕГО ЗА ДЕНЬ`:

```
Отправки:
- 2 человека → Квартал 5, Клетка 2 — мульчирование
- Сидоров, Петров → помощь соседней бригаде
- 3 человека → ремонт инвентаря
```

Записи отправок **не суммируются** с рядами/кустами в `ВСЕГО ЗА ДЕНЬ`.

### 7.4. Отчёт за период (`/period DD.MM-DD.MM`)

> **Критично для MVP.** Прошлая попытка бота провалилась в том числе из-за отсутствия отчёта за период. Этот раздел учитывает **все виды работ** в указанном периоде с разбивкой кто/где/сколько.

```
Период: DD.MM - DD.MM

По видам работ:
- Обрезка: 120 рядов, 8 400 кустов
- Подвязка сухая: 80 рядов, 5 600 кустов
- Опрыскивание: 30 рядов, 2 100 кустов

По сотрудникам:
- Иванов: 50 рядов, 3 500 кустов (обрезка 30, подвязка 20)
- Петров: 45 рядов, 3 100 кустов (обрезка 25, подвязка 20)
- Сидоров: 40 рядов, 2 800 кустов (опрыскивание 30, обрезка 10)

По клеткам:
- Квартал 1, Клетка 3: 60 рядов, 4 200 кустов (обрезка 40, подвязка 20)
- Квартал 1, Клетка 4: 50 рядов, 3 500 кустов (обрезка 30, подвязка 20)
- Квартал 2, Клетка 1: 30 рядов, 2 100 кустов (опрыскивание 30)

Отправки:
- 2 человека → Квартал 5, Клетка 2 — мульчирование (2026-04-25)
- Сидоров, Петров → помощь соседней бригаде (2026-04-26)

ВСЕГО ЗА ПЕРИОД:
Ряды: 230
Кусты: 16 100
```

**Правила:**
- Разбивка по видам работ — обязательна (это была ключевая «дыра» прошлой попытки)
- Отправки выводятся отдельным блоком (см. §4.3) с датой каждой записи
- Если для какого-то сотрудника учитывался режим «только ряды» — кусты по нему = 0 в этих записях, отображаются как «(без кустов)»
- Если данных нет: `Нет данных за указанный период`

---

## 8. Ошибки

При любой ошибке — **остановка обработки** (запись не сохраняется), отправка одного из сообщений:

- `Некорректный формат ввода`
- `Ошибка диапазона рядов` (X > Y, отрицательные числа и т.п.)
- `Ряды отсутствуют в инвентаризации`
- `Обнаружено пересечение рядов`

---

## 9. Хранение данных

**SQLite, схема (черновик):**

- `users` — `user_id`, `tg_id`, `name`, `role` (`admin` / `head_brigadier` / `brigadier`), `head_id` (NULL для admin/head_brigadier; FK на главного для рядового бригадира), `status` (pending/active/blocked), `created_at`

## 10. Импорт инвентаризации

### 10.1. Модель: «у каждого главного бригадира свои кварталы»

В компании несколько **главных бригадиров**; у каждого свой набор кварталов в инвентаризации. Под каждым главным могут работать одна или несколько бригад со своими **рядовыми бригадирами** (см. §2).

В БД таблица `inventory` имеет поле `head_brigadier_id` — каждая запись инвентаризации привязана к **главному**. Когда рядовой бригадир пишет запись учёта, бот:
1. Берёт `head_id` пользователя из `users`
2. Фильтрует инвентаризацию по `head_brigadier_id = users.head_id` (для рядового) или `= users.user_id` (если сам пользователь — главный)
3. Сверяет квартал/клетку/ряд из ввода с этой выборкой

Excel-файлы:
- В одной системе может быть несколько Excel (по периодам посадки или по разным главным)
- Админ при подключении нового главного бригадира загружает его Excel и привязывает кварталы к `head_brigadier_id`
- Команда `/reload-inventory <user_id>` (только админ) — перезагружает инвентаризацию указанного главного бригадира

### 10.2. Источники

Бот поддерживает один или несколько Excel-файлов с инвентаризацией для каждого главного бригадира. Каждый файл может содержать несколько кварталов с разной структурой листов.

Структура записи в БД (после парсинга):

| Поле | Описание |
|------|----------|
| `head_brigadier_id` | FK на главного бригадира |
| `quarter` | Номер квартала |
| `cell` | Номер клетки внутри квартала |
| `row` | Номер ряда внутри клетки |
| `bushes` | Количество кустов в этом ряду |
| `source_file` | Имя исходного Excel (для аудита и `/reload-inventory`) |

> Реальные конкретные данные конкретного главного бригадира (список кварталов, сорта, участки, площади, ФИО) хранятся в `docs/SPEC-private.md` (закрыт через `.gitignore`).

**Парсинг: библиотека `xlsx` (npm). Запуск: один раз при старте бота, либо `/reload-inventory` (только админ).**

**Безопасность:**
- Excel инвентаризации — **секретный документ компании**
- Никогда не коммитить в публичный Git, не выкладывать в интернет, не отправлять в чужие сервисы
- Хранится только: локально + на VPS
- Исключены через `.gitignore` (любые `*.xlsx` в `inventory/`)

**Правила парсинга:**
- Конкретные кварталы, которые вносить или пропускать — определяются для каждого главного бригадира индивидуально (см. `SPEC-private.md`)
- Каждый лист в Excel размечен по-своему (разные заголовки, разное расположение колонок, иногда несколько кварталов на одном листе) — парсер должен обрабатывать листы индивидуально, общего шаблона нет. Конкретные правила парсинга — при реализации Этапа 1.
- Инвентаризация будет меняться: возможны новые кварталы. Перезагрузка через `/reload-inventory` без перезапуска бота
